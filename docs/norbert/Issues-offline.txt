The vertex pointer points to the next (empty) location of the

ojv Functions:
ojvClear();
sets the vertex stack pointer.

ojvSetScale(xyzArray);
sets the aspect ratios, which is used only by ojvCalculate

ojvPushVertex(xyzArray);
pushes a vertex (array containing x, y z) onto the vertex stack

ojvCalculate("operation");
uses the vertex stack to calculate a result  (see table)

ojvPushItem();
pushes all points of selected item onto the vertex stack

ojvVertex(n)
returns a vertex from the stack, where  0 <= n < ojvVertices()

ojvNVertices()
returns the number of vertices in the vertex stack

ojvResultVertex("resultType")
returns a result vertex

ojvResultScalar("resultType")
returns a result scalar

28	ImageJ renaming policy (consequencies for us: see #29):
    open an image that is already open: ImageJ opens it in a second window
    and renames it.

    Renaming detects last minus sign "-" and checks the rest:
    - rest is a digit: it will be incremented*.
    - rest is 1 or 2 letters: they are *discarded* and replaced by a number
    - rest is >2 letters: minus sign plus number is appended

    Renaming is also applied when two different files - with the same
    name but on different locations-  are opened: 
    the second one will be renamed!
 
 	*incremented is only true if lower numbers are occuppied - otherwise,
 	the smallest unoccupied number will be chosen.
 	
 	Example: a user has on disk 10 images im-1 ..im-10;
 	in ImageJ he opens im-1, im-2, im-4.  However, im-4 will
 	open as im-3 if im-4 has already been open.
 	This also happens when opening files via macro.
 	
 	*** This is just a mess!
    > There is no reason to rename images: either you allow same names
    (what most apps do), or you don't (like Object-Image linked files),
    > One image should not be shown in two different windows.
